01 Wall-clock with perf_counter

The whole harness is three ideas: warm up, then bracket the timed loop with two cuda.synchronize() calls. CUDA kernel launches are async — without the syncs you'd time the launch queue, not the work.

import time, torch

def benchmark(fn, *, warmup=5, iters=20):
    for _ in range(warmup):      # kernel autotune, caches, lazy init
        fn()
    torch.cuda.synchronize()     # drain warmup before starting clock
    t0 = time.perf_counter()
    for _ in range(iters):
        fn()
    torch.cuda.synchronize()     # wait for GPU, THEN stop clock
    return (time.perf_counter() - t0) / iters
warmup ×5
sync
measure ×20 → sync → stop

Only the shaded span is timed. The trailing sync is the one people forget.

why perf_counter Monotonic, highest-resolution clock available; unaffected by wall-clock adjustments. Use time.time() and you invite NTP jumps into your measurement.

02 Going deeper with Nsight

perf_counter gives you one number. When you need to know which kernel and why, climb the tool ladder:

  • Annotate with NVTX — wrap regions in torch.cuda.nvtx.range() (or nvtx.annotate) so the timeline is labelled forward / backward / attn instead of a wall of anonymous kernels.
  • nsys (Nsight Systems) — the system timeline: CPU↔GPU overlap, launch gaps, and the CUDA GPU Kernel Summary. Scope that summary to one NVTX range to answer "top kernel in the forward pass."
  • ncu (Nsight Compute) — one kernel, deeply: achieved occupancy, memory vs. compute throughput, and the roofline that tells you whether a kernel is memory-bound or compute-bound.
the ladder perf_counter → is it slow? · nsys → what's slow & where's the bubble? · ncu → why is this one kernel slow?

  • Forward bottleneck: sm80_xmma_gemm_f32f32_f32f32 at 36.9% — the projection/FFN matmuls.
  • Backward bottleneck: cutlass…sgemm_128x128_nt_align1 at 21.6%. Different kernel because backward does ~2 matmuls per linear with transposed operands (nt) → a different GEMM is selected.
  • Elementwise grows from ~8–11% (forward) toward 21% across a full step — all the adds, scales, activation, and gradient plumbing that backward adds on top.

The softmax paradox

Softmax runs in roughly half the time of attn_scores (Q·Kᵀ) or attn_mm (P·V) — yet it does a tiny fraction of their FLOPs:

opFLOPsrelative
attn_scores2·B·H·S²·dcompute-bound
attn_mm2·B·H·S²·dcompute-bound
softmax≈ c·B·H·S²  (no d)memory-bound

Softmax has no d factor — for d in the hundreds it's ~1% of the matmul FLOPs. That it costs half as much runtime means it's bandwidth-bound: it reads and writes the whole S×S score matrix from HBM with almost no arithmetic to hide the latency. This is precisely the motivation for FlashAttention — fuse the passes so the S×S matrix never touches HBM.

04 Mixed precision & autocast

Why the accumulator must stay fp32

Sum 0.01 a thousand times — true answer 10.0. What the accumulator's dtype does to the result:

accumulatoraddendresult
fp32fp3210.0001
fp16fp169.9531
fp32fp16 (implicit)10.0021
fp32fp16 (explicit)10.0021
  • All fp16 → 9.95, off by ~0.5%. Near a sum of ~8, fp16's spacing is ~0.0078; the addend 0.01 barely clears it, so each add rounds away most of its low bits — swamping. Grow the sum a bit more and adds vanish entirely.
  • fp32 accumulator fixes it — no swamping. The small residual (10.0021) is just the fp16-rounded 0.01 (≈0.010002) ×1000, a biased error.
  • Implicit == explicit. s += fp16_tensor already promotes the operand to the accumulator's dtype; the manual .type() is redundant (but documents intent).
the rule & autocast Keep the accumulator in fp32 (master weights, optimizer moments, loss/softmax reductions) even when the operands are low precision. That's exactly what torch.autocast encodes: matmuls drop to fp16/bf16 for tensor-core speed, while softmax / layer_norm / loss stay fp32. Params stay fp32 in storage — autocast casts operands, not storage. bf16 trades mantissa for exponent range, so it avoids the catastrophic swamping without a GradScaler.

autocast, in practice

You don't cast tensors by hand. Wrap the forward + loss in torch.autocast; it applies the per-op policy on the fly. Weights, gradients, and the optimizer stay fp32 — only the ops inside the block change dtype.

# only the forward + loss go inside the context
with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
    logits = model(x)          # matmuls → bf16 (tensor cores)
    loss   = loss_fn(logits, y)  # softmax / loss kept in fp32

loss.backward()                 # grads flow in the dtypes recorded above
optimizer.step()                 # fp32 master weights updated
optimizer.zero_grad()

Two gotchas worth remembering: don't wrap backward() (it replays the recorded dtypes), and for dtype=torch.float16 you also need a GradScaler to stop small grads underflowing — bfloat16 skips the scaler thanks to its wider exponent range.

Concrete: bf16 vs fp32 (per-step, ms)

Same training step, --dtype bfloat16 against the fp32 default. The compute-heavy passes get faster; the speedup grows with model size, and the optimizer step barely moves:

modelpassfp32bf16speedup

  • Speedup scales with size (1.44× → 2.64× on forward): bigger matmuls are more compute-bound, so more of the work lands on tensor cores that run bf16 at higher throughput.
  • Optimizer is flat (1.00×). The step runs on fp32 master weights and is memory-bandwidth-bound — lower-precision compute buys it nothing. This is the "keep the accumulator in fp32" rule showing up in the timing.
  • Watch the variance. Large-model bf16 has high run-to-run spread (forward ±6.58 ms vs ±0.1 ms for fp32) — average over several iterations before trusting a single number.

05 Memory in one training step

Where does the memory go across forward → backward → optimizer? Four buckets — two persistent, two transient. This steps through a tiny 2 → 2 → 2 net with cross-entropy, counting every tensor in floats.

Parameterspersistent

Weights + biases. Fixed by architecture; live the whole run.

Optimizer statepersistent

AdamW keeps two fp32 buffers per param (m, v) → 2×P.

Activationstransient · fwd

Saved in forward because backward needs them; freed as backward consumes them.

reading the sawtooth Params + Adam state are the resident floor — 3×P before a single activation exists, and only an optimizer change (8-bit, sharding) moves it. Activations are the per-step sawtooth you actually control: checkpointing recomputes them, bf16 halves their bytes, FlashAttention keeps attention's S×S block off HBM. This toy peaks at end-of-backward because it's param-heavy (grads 12 > activations 8); real transformers invert that — activations scale with batch×seq×layers, so the peak shifts to the forward→backward boundary, when every activation is still live.

Measure it the way you time things: torch.cuda.max_memory_allocated() for the peak, and the memory-snapshot tool for which tensor and which line caused it.
